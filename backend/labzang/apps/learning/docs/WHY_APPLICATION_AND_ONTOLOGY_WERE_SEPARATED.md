# application과 ontology를 분리했던 이유 (과거)

## 왜 두 개로 나뉘었나

- **ontology**를 learning 루트에 둔 이유는, “LangChain 스타 토폴로지”를 **기존 헥사고날에 붙이는 추가 모듈**로만 생각했기 때문입니다.
- 그래서 **헥사고날 계층(domain / application / adapter)과 별도**로, 같은 레벨에 `ontology/`(hub, spoke, model)를 두었습니다.
- 즉, **아키텍처 원칙보다는 “RAG/스타 토폴로지를 한 덩어리로 넣을 자리”**를 만든 것이고, domain·application과 역할을 나누어 배치한 것은 아닙니다.

## 왜 이렇게 하면 안 좋은가

- **온톨로지 스키마**(지식 그래프 규격)는 **비즈니스 규칙/개념**에 가깝기 때문에 **domain**에 두는 편이 맞습니다.
- **허브·스포크**(어떤 경로로 보낼지, RAG/타이타닉 등 전용 로직)는 **오케스트레이션·정책**이므로 **application**에 두는 편이 맞습니다.
- 그래서 `application`과 `ontology`를 **서로 다른 최상위 폴더**로 두면:
  - “스키마(규격)”와 “라우팅/전용 로직”이 헥사고날 계층과 매핑되지 않고,
  - 의존성 방향(domain ← application ← adapter)이 흐려집니다.

## 정리

- **분리했던 이유**: 스타 토폴로지를 한 덩어리로 넣기 위해, 기존 폴더와 같은 레벨에 `ontology/`를 둔 **실용적 선택**이었음.
- **지금 선택**:  
  - **domain/ontology** = 온톨로지 스키마(지식 그래프 규격)만.  
  - **application/hub**, **application/spokes** = 스타 토폴로지의 허브·스포크.  
  → 이렇게 하면 **헥사고날 계층과 역할이 맞고**, application과 ontology를 “분리한 까닭”도 “같은 레벨에 둔 과거 선택”으로만 설명하면 됩니다.
